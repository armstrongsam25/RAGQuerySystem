# Feature Specification: PDF Upload from the Web UI

**Feature Branch**: `003-pdf-upload-ui`
**Created**: 2026-05-12
**Status**: Draft
**Input**: User description: "Allow the user to upload a PDF to the RAG database from the frontent UI. When they press submit to upload the pdf ask them if they would like to clear the current database and embed on this new pdf or add this pdf to the existing knowledge base. Then automatically do these actions based on what the user responds."

## Overview

Today, a reviewer can ask questions through the UI but can only seed the corpus through the terminal (`rag ingest <path>`). That works for the canned demo, but it breaks down the moment a reviewer wants to point the system at their own PDF — they have to leave the browser, find the file, run a CLI command, then come back. This feature closes the gap: the same page that accepts questions also accepts a PDF upload, asks the reviewer whether the upload should replace or extend the existing corpus, and performs the chosen action without dropping back to the terminal.

The flow is deliberately two-step. The reviewer picks a file and submits. Before the system embeds anything, it asks the reviewer to choose between **replace** (clear the current corpus and ingest only the new PDF) and **append** (add the new PDF to whatever is already ingested). Once the reviewer chooses, the system performs the action and reports the outcome on the same page. Querying remains exactly as before — this feature adds an upload path; it does not change how answers are produced or cited.

What is in scope: a file-picker control on the existing page, a confirmation prompt when the corpus is non-empty, server-side embedding of the uploaded PDF, a "clear corpus" pathway that wipes prior `chunk` and `source_document` rows, and the existing UI states for progress and outcome.

What is out of scope: multi-file uploads, drag-and-drop UX, ingest progress streaming with per-page granularity, per-document filtering at query time, undo for a clear operation, authentication, and any change to the query/citation path.

## Clarifications

### Session 2026-05-12

- Q: On a failed **replace** (extraction, embedding, or persistence failure after the clear step has logically begun), what state must the corpus end in? → A: **Strict rollback** — the prior corpus MUST be fully restored. Implementation will wrap the clear-then-ingest in a single transactional boundary so any failure rolls back the entire operation. Rationale: matches Article I's grounding-safety stance — the system MUST NOT end in a state where queries could silently return less-grounded answers than before the upload was attempted. Trades a slightly longer transaction for a cleaner failure-mode contract.
- Q: What is the default maximum upload size, and how is it surfaced? → A: **25 MB** default, configurable via environment variable (e.g., `RAG_MAX_UPLOAD_BYTES`). Rationale: comfortably accommodates the SC-001 50-page-PDF ceiling with figures, prevents accidental denial-of-disk on a local-dev stack, and matches typical assessment-PDF size ranges. The exact env-var name is an implementation-level decision for `/speckit-plan`; the default value is pinned here.
- Q: How is the replace-or-append confirmation surfaced in the UI? → A: **Inline two-step** within the existing HTMX response region. The first POST (file submit) returns a confirmation partial rendered into the response area, showing Replace, Append, and Cancel controls. Each control issues a second POST to the upload endpoint with the chosen action. Rationale: aligns with Feature 002's server-rendered HTMX partial pattern (the response area already swaps `_answered.html`/`_refused.html`); no modal library, no client-side state, no extra route. The two-POST shape also makes the server-side flow uniform — the second POST carries `action=replace|append` per FR-013, regardless of whether the first POST was on an empty corpus (skipped confirm) or a populated one.
- Q: How does the system handle a second upload submitted while one is already in flight? → A: **Hard-reject with 409 Conflict** — the second upload is rejected immediately with a "another upload is in progress, please retry" error rendered in the UI's error panel. No server-side queuing, no waiting. Rationale: single-user local-dev demo has no SLA to honor; immediate feedback beats opaque blocking, server semantics stay simple (single in-process flag or advisory lock), and there is zero risk of queue-induced HTTP timeouts. Defense in depth: the UI SHOULD also disable the upload control on the originating page while a request is in flight (consistent with FR-003's work-in-progress indicator), but the server reject is the authoritative backstop for the multi-tab case.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Upload a PDF and Replace the Corpus (Priority: P1)

A reviewer has the stack up and a previously-ingested PDF in the database. They want to point the system at a different PDF and start fresh — they don't want the new PDF's answers contaminated by chunks from the old one. They open the UI, pick their PDF in the upload control, and submit. The UI tells them the corpus already has content and asks: replace it, or add to it? They choose **replace**. The system clears the existing `source_document` and `chunk` rows, ingests the uploaded PDF (extract → chunk → embed → persist), and shows a success message naming the new document and the chunk count. They can immediately ask a question and get answers grounded only in the new PDF.

**Why this priority**: This is the headline behavior the user asked for and the most common reviewer flow during a live demo — "let me try with my own document." Without it, the upload UI exists but the reviewer's mental model ("I'm starting over") isn't supported, and stale chunks from the previous PDF would silently bleed into answers, breaking the grounding signal. P1 because the demo's credibility depends on a clean reset being one click away.

**Independent Test**: With the stack up and at least one `source_document` row and one `chunk` row present, open the UI, upload a different PDF, and on the confirmation prompt select **replace**. Verify (a) prior `source_document` and `chunk` rows are gone, (b) new rows exist for the uploaded PDF with all Article II provenance fields populated, (c) the UI shows a success message naming the new document, and (d) a query for a fact known only to the *prior* PDF returns `no_documents` or a refusal — never a hallucinated answer drawn from chunks that should have been deleted.

**Acceptance Scenarios**:

1. **Given** the corpus already contains at least one ingested PDF, **When** the reviewer uploads a new PDF and selects **replace** at the confirmation prompt, **Then** all prior `source_document` and `chunk` rows are removed before any new content is persisted, the new PDF is ingested with all Article II provenance fields populated, and the UI shows a success message naming the new document and the resulting chunk count.
2. **Given** a **replace** is in progress, **When** the system fails partway (e.g., the embedding model is unreachable, persistence errors out), **Then** the database MUST end in the same state it was in before the upload — the prior corpus is fully preserved (strict rollback per FR-019). The UI MUST surface the failure cause AND confirm that the prior corpus is intact so the reviewer can retry without having to verify state through a query.
3. **Given** a successful **replace**, **When** the reviewer asks a question whose answer was only in the *previous* PDF, **Then** the response is `refused` or `no_documents` — never an answered response citing chunks that were supposed to be deleted.

---

### User Story 2 - Upload a PDF and Append to the Corpus (Priority: P1)

A reviewer wants to grow the knowledge base across multiple PDFs rather than swap between them — for example, ingesting a related second document so questions about either are answerable. They open the UI, pick a new PDF, and submit. The UI tells them the corpus already has content and asks: replace, or add? They choose **append**. The system ingests the new PDF *in addition to* whatever is already there. Prior `source_document` rows and their chunks remain untouched. A subsequent query can return citations from either document, and the citation page numbers continue to resolve correctly back to the right PDF.

**Why this priority**: Article II of the constitution requires every citation to point back to a verifiable source. Append is what makes the multi-document case demonstrable end-to-end — without it, "the schema already supports multiple source documents" (Feature 002 assumption) is unverified in the demo. P1 because append and replace are the two halves of the same user choice; shipping only one would force the reviewer to guess the system's behavior.

**Independent Test**: With one PDF already ingested, upload a second, distinct PDF and select **append**. Verify (a) the original `source_document` row and its `chunk` rows are intact, (b) a new `source_document` row exists for the uploaded PDF with its own chunk rows, (c) a query whose answer is in the *original* PDF still produces an answered response with citations resolving to the original PDF's page numbers, and (d) a query whose answer is in the *newly-uploaded* PDF produces an answered response with citations resolving to that PDF's page numbers — neither query's citations point to the wrong document.

**Acceptance Scenarios**:

1. **Given** the corpus already contains an ingested PDF, **When** the reviewer uploads a different PDF and selects **append**, **Then** prior `source_document` and `chunk` rows are preserved unchanged, a new `source_document` row is created for the uploaded PDF with its own chunk rows fully populated, and the UI shows a success message naming the new document and the resulting chunk count.
2. **Given** the reviewer attempts to **append** a PDF whose content was already ingested under the same identity (same chunks already exist per the FR-004 dedup rule), **When** the upload runs, **Then** no duplicate chunks are created and the UI surfaces a clear "no new content" message — distinct from a generic success — so the reviewer understands their upload was a no-op.
3. **Given** two PDFs have been ingested via successive **append** operations, **When** the reviewer asks a question answerable from either document, **Then** every citation in the response carries the correct page number for the document it came from, and a reviewer opening the named PDF to the cited page can find the quoted text verbatim.

### User Story 3 - Upload Into an Empty Corpus (Priority: P2)

A reviewer is using the stack for the first time after `make up`. No PDFs have been ingested. They open the UI, see that questions return a "no documents ingested" hint, and use the upload control to ingest their first PDF. Because the corpus is empty, the replace/append confirmation is unnecessary and the system skips it — the upload proceeds directly. On success, the UI shows the document name and chunk count, and the reviewer can immediately ask questions.

**Why this priority**: P2 because the empty-corpus case is real but it's a one-time experience per stack stand-up — most reviewer time is spent in the "corpus already has content" world that US1 and US2 cover. The same code paths from US1/US2 handle this case with the confirmation step skipped; treating it as a separate P1 would overstate its weight in the design.

**Independent Test**: On a stack with zero `source_document` rows, open the UI, upload a PDF, submit. Verify (a) no confirmation prompt is shown (since there is nothing to replace and nothing to append to), (b) the upload proceeds and ingests the PDF, (c) the success message names the document and chunk count, and (d) the UI's question form transitions from "no documents ingested" hint to normal queryable state without requiring a page reload.

**Acceptance Scenarios**:

1. **Given** the corpus is empty, **When** the reviewer uploads a PDF and submits, **Then** no replace-or-append confirmation is shown and the PDF is ingested directly, producing a populated `source_document` row and chunk rows with all Article II provenance fields.
2. **Given** the upload into an empty corpus succeeded, **When** the reviewer submits a question whose answer is in the uploaded PDF, **Then** the response is answered with at least one citation — no page reload was required between upload-success and the first answerable query.

### Edge Cases

- **Upload of a non-PDF file** (e.g., a `.docx` or a renamed binary): The UI MUST reject the file at the boundary with a clear message naming the violation. The server MUST NOT attempt extraction or embedding. The corpus MUST be untouched. (The client may use the file-picker's `accept` attribute as a first filter, but the server MUST also validate — the client filter is a UX hint, not a security boundary.)
- **Upload of a PDF that exceeds the configured maximum size**: The server MUST reject with a clear message naming the size limit. The upload MUST NOT consume disk or memory beyond the configured cap. The size cap MUST be configurable and surface in the rejection message.
- **Upload of a PDF with no extractable text** (corrupt, password-protected, or scanned-image-only): Ingest MUST fail with an actionable error, the UI MUST surface that error, and the corpus MUST be left in the same state it was in before the upload — no half-written `source_document`, no orphaned chunks.
- **Reviewer closes the browser tab mid-upload**: The server MUST complete or cleanly abort the in-flight operation; the database MUST end up in a queryable state (either fully ingested, or fully rolled back to the pre-upload corpus for the chosen action). The reviewer reopening the page MUST see the actual current state of the corpus, not a stale "uploading…" indicator.
- **Two reviewers (or two tabs) upload at the same time**: Per the 2026-05-12 clarification, the system MUST **hard-reject** the second upload with an HTTP 409 Conflict and an error message naming the in-progress upload. The UI's error panel MUST render the rejection so the reviewer can retry once the in-flight upload completes. The system MUST NOT queue, serialize, or interleave concurrent uploads, and MUST NOT allow concurrent **replace** operations under any circumstance.
- **Reviewer chooses "replace" by mistake on a corpus they didn't mean to clear**: The confirmation step is the protection. Once the reviewer confirms **replace**, the prior corpus is gone — there is no undo. The confirmation prompt MUST make this consequence explicit (e.g., "This will delete all currently-ingested documents") so the choice is informed.
- **Upload of a PDF whose chunks already match existing chunks under the FR-004 dedup rule, via append**: No new chunks are created. The UI MUST report this distinctly from "new chunks were created" so the reviewer understands why the chunk count didn't grow.
- **Upload of a PDF via replace when only the to-be-uploaded PDF is already ingested**: Replace MUST still clear the existing corpus and re-ingest from the uploaded copy. (The point of replace is "I want a clean slate from this file" — preserving the existing rows because the bytes look familiar would surprise the reviewer.)
- **Large PDF that takes longer than the typical query latency to ingest**: The UI MUST keep the reviewer informed that work is in progress (the existing HTMX indicator pattern from Feature 002 is sufficient — no per-page progress is required). The reviewer MUST NOT be allowed to submit a query against the partially-populated corpus until ingest reports success or failure.

## Requirements *(mandatory)*

### Functional Requirements

#### Upload UI

- **FR-001**: The existing UI page (the one that already serves the question form per Feature 002 FR-017/FR-018) MUST present a PDF upload control alongside the question form. The control MUST accept exactly one PDF file at a time.
- **FR-002**: The upload control MUST be visually and structurally distinct from the question form. A reviewer MUST be able to tell at a glance which control is for asking and which is for adding documents.
- **FR-003**: When the reviewer submits an upload, the UI MUST indicate work-in-progress in the same way the existing query path does (per Feature 002 FR-021). The reviewer MUST NOT be left wondering whether the upload registered.
- **FR-004**: After an upload succeeds, the UI MUST display a result panel naming the ingested document (filename), the resulting chunk count for that document, and which action (replace or append) was performed. The result panel MUST be visually distinct from query answers and refusals so a reviewer cannot confuse an upload outcome with a query outcome.
- **FR-005**: After an upload fails, the UI MUST display an error panel that names the cause in user-readable terms (e.g., "this file is not a PDF", "this PDF could not be read", "ingest failed during embedding — please retry"). The error panel MUST be visually distinct from refusals.
- **FR-006**: Submitting an upload MUST NOT cause the UI to lose unrelated state. In particular, if the reviewer had a query response on the page when they uploaded, the upload result MUST render in its own region rather than overwriting the answer (or, equivalently, the page may render uploads in the same response region but with an unambiguous label distinguishing it from query answers — the choice is implementation-level; the requirement is that the reviewer can tell which is which).
- **FR-007**: The UI MUST NOT include analytics, telemetry, third-party trackers, or any external network calls beyond the request to the local backend API. (Carried forward from Feature 002 FR-022.)

#### Replace-or-Append Confirmation

- **FR-008**: When the reviewer submits an upload AND the corpus is non-empty (at least one `source_document` row exists), the system MUST present a confirmation step asking the reviewer to choose between **replace** (clear the existing corpus, then ingest the uploaded PDF as the only document) and **append** (preserve the existing corpus, ingest the uploaded PDF in addition). Per the 2026-05-12 clarification, the confirmation is rendered **inline** in the existing HTMX response region as a server-rendered partial — not as a modal overlay, separate page, or browser-native dialog. The partial MUST present Replace, Append, and Cancel controls, each of which issues a follow-up request to the upload endpoint carrying the chosen action.
- **FR-009**: The confirmation step MUST display, in plain language, the consequence of each option — at minimum, that **replace** will delete all currently-ingested documents and cannot be undone, and that **append** will add to the existing corpus without removing anything.
- **FR-010**: The confirmation step MUST NOT execute either action until the reviewer makes an explicit choice. There MUST be no default that fires after a timeout; the action MUST be triggered by the reviewer's affirmative selection.
- **FR-011**: When the corpus is empty (zero `source_document` rows), the system MUST skip the confirmation step and proceed directly with the upload — there is nothing to replace and nothing to append to, so asking would be friction.
- **FR-012**: The reviewer MUST be able to cancel the confirmation step (e.g., a cancel control) and have the upload abort with no side effects on the database. After cancel, the corpus MUST be exactly as it was before the upload was submitted, and the upload control MUST be ready to accept another file.

#### Server-Side Upload Handling

- **FR-013**: The backend MUST expose an HTTP endpoint that accepts a multipart file upload from the UI. The endpoint MUST accept the chosen action (**replace** or **append**) as part of the request so the server does not have to remember per-session state between the file submission and the confirmation choice. Per the 2026-05-12 inline-confirmation clarification, the flow is two requests: (a) the initial POST carrying the file (the server replies with either a confirmation partial when the corpus is non-empty, or the ingest result directly when the corpus is empty); (b) when a confirmation was returned, a follow-up POST carrying the file again together with the chosen action. The server MUST NOT rely on per-session storage of the uploaded bytes between the two requests — the file MUST be present in the second POST, and the server MUST re-validate it (FR-014, FR-015) before any database mutation.
- **FR-014**: The backend MUST validate the uploaded file is a PDF before any database mutation occurs. Validation MUST NOT rely solely on the filename extension — it MUST inspect the file's content (e.g., the PDF magic header) so a renamed non-PDF cannot reach the extractor.
- **FR-015**: The backend MUST enforce a configurable maximum upload size. The default cap MUST be **25 MB** (per the 2026-05-12 clarification) and MUST be overridable via an environment variable. Requests exceeding the configured cap MUST be rejected with an HTTP 413 (or equivalent payload-too-large status) and an error payload naming the cap in bytes AND in megabytes for reviewer-readable rejection messages. The rejection MUST occur before any extraction or embedding work is attempted.
- **FR-016**: When the action is **replace**, the backend MUST clear the existing `source_document` and `chunk` rows BEFORE persisting any new chunks. The clearing MUST cascade so no `chunk` rows are left orphaned without their parent `source_document`. The replace operation MUST be ordered such that a concurrent query never observes a half-cleared corpus.
- **FR-017**: When the action is **append**, the backend MUST NOT touch any existing `source_document` or `chunk` rows. The uploaded PDF MUST be ingested using the same code path as `rag ingest <path>` (Feature 002 FR-001 through FR-006) — chunking, embedding, and persistence MUST be identical, so an uploaded PDF and a CLI-ingested PDF are indistinguishable in the database.
- **FR-018**: The Article II dedup invariant (Feature 002 FR-004) MUST hold for uploaded PDFs the same way it holds for CLI-ingested PDFs. Re-uploading the same PDF via **append** MUST NOT create duplicate chunks; the response MUST surface "no new content" so the UI can render the FR-004 distinct success message.
- **FR-019**: An upload that fails (validation rejection, extraction failure, embedding failure, persistence failure) MUST leave the database in a state where the reviewer can re-upload cleanly without manual cleanup. For **replace**, the prior corpus MUST be fully preserved (per the 2026-05-12 strict-rollback clarification): the clear-step and the ingest MUST execute inside the same transactional boundary, and any failure inside that boundary MUST roll back both the clear and any partial inserts so the corpus is observably identical to its pre-upload state. For **append**, no chunks from the failed upload may be committed; the prior corpus is untouched by definition. In all failure modes, the implementation MUST NOT leave half-written `source_document` rows or orphaned chunks.
- **FR-027**: After a **replace** failure rolls back per FR-019, the upload response MUST surface the failure cause AND explicitly confirm that the prior corpus is intact (e.g., "Upload failed: <cause>. Your existing documents are unchanged."). This lets the reviewer retry without first having to verify corpus state through a query.
- **FR-028**: While an upload is in flight (between the server receiving the file and the response — whether confirmation partial, success, or failure — being sent), the backend MUST reject any additional upload POST with an HTTP 409 Conflict and an error payload naming the in-progress upload's start time. The mechanism MUST be process-wide (not per-request-handler) so concurrent requests across worker threads are caught. The reject MUST occur before any validation, extraction, or database work on the second upload. The UI SHOULD additionally disable its upload submit control while a request is in flight as a UX courtesy, but the server reject is the authoritative guarantee.
- **FR-020**: The upload endpoint MUST emit structured log records for at minimum: upload received (with filename and reported content type), validation outcome (accepted or rejected with cause), action chosen (replace or append), clear outcome (when replace), ingest start/end (per Feature 002 FR-005 logging contract), and the final upload outcome with elapsed time.
- **FR-021**: The uploaded PDF MUST NOT be persisted to disk outside the database beyond what is required to perform the extraction step. (Carried forward from Feature 002 FR-026 — no caching of full PDFs to the filesystem after the request completes.)

#### Cross-Cutting

- **FR-022**: The CLI `rag ingest <path>` workflow MUST continue to work unchanged. Adding the upload UI MUST NOT break the demo's existing terminal flow (Feature 002 SC-006 — clone-to-answer in under 10 minutes from the README).
- **FR-023**: The query endpoint and citation contract (Feature 002 FR-007 through FR-016) MUST NOT change as a result of this feature. Citations from uploaded PDFs MUST be indistinguishable in structure from citations from CLI-ingested PDFs.
- **FR-024**: All new public functions MUST carry type annotations and pass the linter checks the codebase already enforces (carried forward from Feature 002 FR-023).
- **FR-025**: Test coverage MUST include, at minimum: a successful **replace** flow (verified by chunk-count delta and a query that previously answered now refusing), a successful **append** flow (verified by chunk-count delta and queries against both source documents resolving correctly), the empty-corpus upload path (no confirmation, direct ingest), the cancel-confirmation path (database unchanged), the non-PDF rejection path (validation before any database mutation), and the oversize-file rejection path.
- **FR-026**: The README MUST document the upload flow at the same level of detail as the existing CLI flow, including the replace-vs-append choice and its consequences.

### Key Entities *(include if feature involves data)*

The persistence-layer entities (`source_document`, `chunk`) are inherited from Feature 001 and unchanged. This feature introduces the following **runtime** entities that flow through the upload API:

- **Upload Request**: The reviewer's uploaded PDF together with the chosen action. Carries the file bytes, the original filename (used only for display and logging — never for dedup; dedup uses chunk-level identity per FR-018), the reported content type, and the action selector (`replace` or `append`).
- **Upload Response**: The system's reply after the upload completes. Carries an outcome (`succeeded`, `no_new_content`, `failed`), the document identifier and chunk count when succeeded, the cause when failed (`invalid_pdf`, `oversize`, `extraction_failed`, `embedding_failed`, `persistence_failed`, `concurrent_upload`), and the action that was performed (so the UI can render the FR-004 success panel).
- **Corpus State**: Not a persisted entity, but a derived view of "how many `source_document` rows currently exist." Used by the UI to decide whether to present the confirmation step (FR-008) or skip it (FR-011). MUST reflect the database's current state — not a cached snapshot — at the moment the upload is submitted.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A reviewer with the stack up can upload a PDF through the UI, choose **replace**, and have a queryable corpus (the new PDF only) in under 3 minutes for a PDF of up to 50 pages — matching the SC-001 budget from Feature 002 for CLI ingest, since the underlying ingest path is the same.
- **SC-002**: A reviewer choosing **append** for a second PDF receives a success outcome in the UI, and a subsequent query against either source document returns an answered response with citations that resolve correctly to the right document, in 100% of a hand-curated sample of 5 in-scope questions (at least 2 per document).
- **SC-003**: On a successful **replace**, a query for a fact known only to the *prior* corpus returns `refused` or `no_documents` in 100% of attempts on a hand-curated sample of 3 such questions — no hallucinated answer is ever produced from chunks that were supposed to be deleted.
- **SC-004**: An attempted upload of a non-PDF file is rejected with a clear error in the UI in 100% of attempts, and the corpus row counts (`source_document`, `chunk`) MUST be observably unchanged before and after every rejection.
- **SC-005**: A reviewer canceling the replace-or-append confirmation step ends in a database state identical to the pre-upload state (verified by row-count equality) in 100% of attempts.
- **SC-006**: A reviewer on a fresh clone, with the stack up and `data/sample.pdf` not yet ingested, can complete an upload through the UI (empty-corpus path) and get a rendered answer to a question in the same browser session in under 5 minutes — without ever touching the terminal after `make up`.
- **SC-007**: The UI surfaces a distinct visual state for each of: upload success, upload failure, query answered, query refused, and query empty-corpus. A reviewer correctly identifies which case occurred 100% of the time across a sample of at least 10 responses spanning all five states.

## Assumptions

- **No change to the embedding or generation models**: This feature reuses the Feature 002 ingest pipeline (chunking, embedding, persistence) verbatim. Constitution Article IV's stack pins are inherited; no new model choices are made here.
- **No change to the query path**: The query endpoint and citation contract (Feature 002 FR-007 through FR-016) are unchanged. Citations from uploaded PDFs flow through the same retrieval, grounding, and rendering paths as citations from CLI-ingested PDFs.
- **Upload UI is delivered as HTMX route(s) on the existing FastAPI app**: Consistent with the Feature 002 clarification that kept the compose topology at two services (`app` + `db`). No additional service is introduced — the upload endpoint lives in the same code path as the query endpoint.
- **Single-user, local-development trust boundary**: The stack remains unauthenticated and assumes a local-development trust boundary, consistent with Feature 002's auth-out-of-scope assumption. There is no per-user corpus; "the corpus" is the contents of the `chunk` table.
- **Single-file upload, single-tab assumption for the typical demo flow**: The system MUST handle the concurrent-upload edge case (per the Edge Cases section), but the *expected* demo flow is one reviewer in one tab uploading one PDF at a time. Multi-file uploads, drag-and-drop, and ingest-progress streaming are explicitly out of scope.
- **Replace is destructive and irreversible**: A confirmed **replace** deletes the prior corpus permanently. There is no undo, no snapshot, no soft-delete. The confirmation step's wording is the only safeguard — FR-009 holds the line that the consequence is communicated before the reviewer confirms.
- **The ingest code path is the single source of truth for chunking and embedding**: The upload endpoint MUST call into the same ingest implementation that `rag ingest <path>` calls. Any divergence would mean a PDF could behave differently depending on how it was loaded — the constitution's grounding signal would degrade silently.
- **Dedup is content-based, not filename-based**: Per Feature 002 FR-004, dedup uses the UNIQUE constraint on (source document, page, char offsets). Two uploads of the *same* PDF (under any filename) via **append** will produce zero new chunks. This is correct behavior, surfaced to the UI per FR-018.
- **Maximum upload size default**: Per the 2026-05-12 clarification, the default cap is **25 MB**, configurable via environment variable. This sizing comfortably accommodates the SC-001 50-page-PDF ceiling (including PDFs with embedded figures and fonts), keeps the local-dev stack safe from accidental denial-of-disk, and aligns with typical assessment-PDF size ranges. Reviewers with larger documents can raise the cap without code changes.
- **Browser-supported PDF content types**: The upload accepts files whose content-type the browser reports as `application/pdf` and whose first bytes match the PDF magic header. Edge cases like `application/octet-stream` from older browsers MAY be accepted IF the magic-header check passes — the magic-header check is authoritative, not the reported content type.
- **No streaming responses**: Upload responses are returned whole, consistent with constitution Article VII's no-streaming stance. The UI's in-progress indicator (FR-003) does the job of keeping the reviewer informed while the request is open.
