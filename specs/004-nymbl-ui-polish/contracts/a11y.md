# Contract: Accessibility Attributes

Implements FR-019 and FR-020 (the 2026-05-13 clarify decision on assistive tech) plus FR-014 (WCAG AA contrast) and FR-016 (`prefers-reduced-motion`).

## ARIA live regions

The following elements MUST carry the listed attributes after server-side rendering:

| Element | Attribute | Value | Source |
|---|---|---|---|
| `<section id="response">` in [base.html](../../../src/rag/ui/templates/base.html) | `aria-live` | `"polite"` | FR-019 |
| `<div id="current-doc">` in [base.html](../../../src/rag/ui/templates/base.html) | `aria-live` | `"polite"` | FR-019 |
| `.upload-progress` root in [_upload_in_progress.html](../../../src/rag/ui/templates/_upload_in_progress.html) | `aria-live` | `"polite"` | FR-019 |
| `.status-error` root in [_error.html](../../../src/rag/ui/templates/_error.html) | `role` | `"alert"` | FR-020 |
| `.status-upload-error` root in [_upload_error.html](../../../src/rag/ui/templates/_upload_error.html) | `role` | `"alert"` | FR-020 |
| The client-side network/timeout error `<template>` block | `role` | `"alert"` | FR-020 |

Note: `role="alert"` implies `aria-live="assertive"` and `aria-atomic="true"`. We MUST NOT *also* set `aria-live` on the same element — that produces conflicting hints in some screen readers (NVDA in particular).

## Reduced motion

The pulse animation defined for `.htmx-indicator` and the staged upload progress dot MUST be wrapped in:

```css
@media (prefers-reduced-motion: reduce) {
  .htmx-request .htmx-indicator,
  .htmx-request.htmx-indicator,
  .upload-progress-stage--active .upload-progress-stage-dot {
    animation: none;
    opacity: 1;
  }
}
```

This MUST come *after* the original `@keyframes` and `animation:` declarations so it overrides them via cascade. Reduced-motion users still see "Thinking…" and the active-stage label — only the animation is suppressed.

## Contrast targets (WCAG AA)

The contract test runs no automated contrast calculation (those are brittle and require parsing CSS variables fully); instead it asserts the *pairings* implied by the brand guide are not violated. The full pairing matrix is documented in [nymbl-brand.md §2.5](../../../nymbl-brand.md). For this feature:

| Pairing in our UI | Contrast | Status |
|---|---|---|
| `--ink-900` text on `--paper-50` body | ~17.8:1 | PASS |
| `--stone-500` (muted) on `--paper-50` | ~4.7:1 | PASS for body |
| `--ink-900` on `--signal-500` (the Ask button) | ~14.2:1 | PASS |
| `--signal-600` on `--paper-50` (the pulsing indicator text) | ~3.6:1 | PASS at body-lg (18px / 500 weight) only |
| `--paper-50` on `--ink-900` (dark mode body) | ~17.8:1 | PASS |
| `--stone-300` on `--ink-900` (dark mode muted) | ~6.8:1 | PASS |

The forbidden pairing `--paper-50` on `--signal-500` (1.3:1) MUST NOT appear anywhere — the contract test scans the stylesheet for adjacent declarations that produce this combination by looking for any selector that sets both `color: var(--paper-50)` and `background: var(--signal-500)` (or any swap thereof). The check is heuristic but catches the obvious mistakes.

## Keyboard focus

The existing focus styles (`box-shadow: 0 0 0 3px rgba(41, 98, 255, 0.15)` on `.btn-ask`, etc.) MUST be migrated to use the brand accent. The replacement is `box-shadow: 0 0 0 3px color-mix(in srgb, var(--signal-500) 35%, transparent)`. The contract test does not enforce this specifically (too easy to false-positive), but the manual quickstart checklist covers it.

## Screen reader regression test

The contract test renders the base template and each error partial and asserts the attributes from the table above are present. The test does not exercise an actual screen reader (out of scope for unit tests) — manual verification with VoiceOver / NVDA is captured in the quickstart.
